## Context

Phase 4 requires persisting exercise information in the unified database. Three design questions arise:

1. **Which database?** ADR-0003 established a hybrid model: GlobalBase for reference data, LocalBase for project-specific data.

2. **What goes in the DB vs. what stays in `.tex` files?** Exercise content (stems, options, solutions) is authored and maintained in LaTeX. Storing content in the DB creates a synchronization problem ‚Äî the `.tex` file is always the truth source.

3. **Where do `.tex` files live?** All exercise files live in the centralized `00EE-ExamplesExercises` directory. Individual projects (courses, exams) import specific exercises via symlinks or `\input` paths.

### File Lifecycle

`crete` creates placeholder `.tex` files from book references. These evolve through stages:
- **placeholder** ‚Äî skeleton generated by `crete`, no content yet
- **in_progress** ‚Äî author is writing the solution/options
- **complete** ‚Äî ready for use in evaluations and Moodle export

The DB must track this status so that `build-exam` and `export-moodle` only use complete exercises.

### Image Handling

Exercise content frequently includes images:
- Stem text may include diagrams, figures, or photographs (`\includegraphics`, `\inputsvg`, TikZ inline)
- Answer options may be graphs or diagrams (choose the correct graph)
- Solutions may include worked diagrams

Images are referenced by path in the `.tex` file. The DB does not store image content but must be aware of image references for Moodle export (where images must be embedded or base64-encoded).

### Options Considered

**Option A: Exercises in GlobalBase only**
- Pro: Single queryable bank
- Con: Exercise files live in project directories; path mismatch

**Option B: Exercises in LocalBase only**
- Pro: Co-located with project
- Con: Cannot query across projects

**Option C: Exercises in GlobalBase with project reference (selected)**
- Pro: Single queryable bank ‚Äî "find all exercises by tag/taxonomy" works globally
- Pro: `build-exam` can select from the full bank without cross-DB joins
- Pro: Mirrors how Content and Book already work in GlobalBase

---

## Decision

**Option C: Exercises in GlobalBase. The `.tex` file is the truth source; the DB stores only metadata and file references.**

### Core Principle: File as Truth

The DB is an **index** over exercise files, not a content store. Exercise text (stems, options, solutions) is read from `.tex` files at parse time and never persisted in the DB. This means:
- No synchronization problem ‚Äî re-parsing always picks up the latest content
- `crete` creates files freely; the DB catches up via `exercise sync`
- Editing an exercise means editing the `.tex` file, not the DB

**What is NOT in the DB:**
- `stem_text` ‚Äî read from `.tex` at parse/export time
- `solution_text` ‚Äî read from `.tex` at parse/export time
- `feedback_text` ‚Äî read from `.tex` at parse/export time
- Option text ‚Äî read from `.tex` at parse/export time

### Synchronization Commands

**`workflow exercise sync [PATH]`** ‚Äî one-time or on-demand full synchronization:
1. Scan `00EE-ExamplesExercises` (or given path) for all `.tex` files
2. Parse commented YAML metadata from each file
3. Upsert Exercise records: insert new, update changed (by `file_hash`), warn on orphans
4. Report: N new, N updated, N unchanged, N orphaned

**`workflow exercise gc`** ‚Äî garbage collect orphaned records:
1. Find Exercise records whose `source_path` no longer exists on disk
2. Prompt for confirmation, then delete

**Cron/periodic sync** ‚Äî optional scheduled job:
- Run `exercise sync` periodically (e.g., daily) to keep the index fresh
- Implemented as a system cron or `workflow` hook, not a daemon

### Relationship to Existing Models

```
Exercise.content_id  ‚Üí Content.id       (which chapter/section this tests)
Exercise.book_id     ‚Üí Book.id          (which textbook)
Exercise.type        ‚âà Item.name        (conceptual alignment, not FK)
Exercise.taxonomy_*  ‚âà Item.taxonomy_*  (same enum values, validated at parse time)

build-exam: SELECT FROM Exercise
            WHERE taxonomy matches EvaluationItem constraints
            AND status = 'complete'
```

`Exercise` does **not** FK to `Item` directly. `Item` is an abstract slot ("2 application-level questions"); `Exercise` is a concrete instance. `build-exam` bridges them by matching taxonomy constraints.

## Architectural Rules

### MUST

- `.tex` files **MUST** be the single truth source for exercise content.
- The DB **MUST NOT** store exercise text (stem, solution, options, feedback).
- `Exercise` and `ExerciseOption` **MUST** use `GlobalBase`.
- `exercise_id` (from YAML) **MUST** be unique across the global bank.
- `status` field **MUST** track file lifecycle: `placeholder`, `in_progress`, `complete`.
- `file_hash` **MUST** be checked during sync to detect changes.
- `exercise sync` **MUST** be idempotent (re-running produces the same result).

### SHOULD

- `source_path` **SHOULD** be absolute (exercises live in a known central location, not per-project).
- Parser **SHOULD** detect image references (`\includegraphics`, `\inputsvg`, `\qdiagram`) and populate `has_images` and `image_refs`.
- `exercise gc` **SHOULD** require explicit confirmation before deleting orphaned records.
- `exercise sync` **SHOULD** report status changes (e.g., "serway-ch01-003: placeholder ‚Üí complete").

### MAY

- A cron job **MAY** run `exercise sync` periodically for hands-free index maintenance.
- A future `ExerciseVersion` table **MAY** track content evolution via `file_hash` history.
- `image_refs` **MAY** later be promoted to a proper M2M table if image management demands it.

---

## Consequences

### Benefits

- No content duplication ‚Äî `.tex` is always authoritative
- DB is lightweight (metadata only) ‚Äî fast queries, small footprint
- `crete` can create placeholder files without worrying about DB consistency
- Status tracking prevents incomplete exercises from appearing in exams/exports
- Image awareness enables Moodle export to handle embedded figures

### Costs

- Export and `build-exam` require file I/O at runtime (read `.tex` files) ‚Äî acceptable, exercises are small files
- Sync must handle file renames/moves (detected via `exercise_id` in YAML, not path)
- Orphan detection requires filesystem access (cannot be done from DB alone)
