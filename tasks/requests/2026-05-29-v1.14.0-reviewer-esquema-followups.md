## Deferred items

### 1. (Phase 1.5) Shared neutral error base for `BibKeyAmbiguous` — MEDIUM
There are now two `BibKeyAmbiguous` classes: bibliography-layer (bare `Exception`)
and content-layer (`ContentServiceError` subclass), with manual string
re-formatting at the boundary in `content/bib_links.py::_resolve_bib_entry`.
Correct layering for now, but Phase 1.5's planned `EntityNotFoundError` /
`UniquenessError` consolidation should introduce a shared neutral base (candidate:
`workflow.db.errors`) that both layers inherit, so the boundary catch becomes a
re-raise (`raise ContentBibKeyAmbiguous(...) from exc`) rather than a re-message.
Files: `src/workflow/content/bib_links.py`, `src/workflow/bibliography/service.py`.

### 2. ADR for the `workflow.bibliography` module boundary — LOW
A new peer module to `content`/`exercise`/`prisma` warrants a one-paragraph ADR:
(a) it is a foundation/lookup layer others depend on; (b) the prisma re-export is a
deprecation path with a removal target; (c) the 0/1/2+ lookup contract. Bundle with
the already-planned ADR-0007 amendment (read-heavy CLI surfaces may bypass repo
protocols).

### 3. Prisma re-export removal milestone — LOW
`prisma/service.py:13` re-exports `get_bib_entry_by_bibkey` for back-compat
(`# noqa: F401`). Add a removal milestone once `exercise/*` (done) and any other
callers import from `workflow.bibliography` directly. Note: `_bib_entry_options`
is a genuine internal prisma dependency, NOT just a re-export — do not remove it.

### 4. `BibEntry.year` test-fixture type consistency — LOW
`BibEntry.year` is `Mapped[int | None]`. New exercise test fixtures pass
`year="2020"` (str) while the bibliography test passes `year=2024` (int). Both pass
under SQLite type affinity but are inconsistent and would fail on a stricter
backend. Align exercise test fixtures to `int`. Files:
`tests/workflow/exercise/test_cli.py`, `tests/workflow/exercise/test_service.py`.

### 5. (from v1.14.1 architect review) Promote generic error bases to `workflow.db.errors` — MEDIUM
v1.14.1 added `EntityNotFoundError` / `UniquenessError` under `ContentServiceError`, but in
`src/workflow/content/service.py` — a leaf feature module. They are domain-agnostic; the
moment `exercise`/`prisma`/`bibliography` want them they would have to import *upward* from a
sibling feature (the coupling inversion ADR-0007 guards against). When the deferred neutral-base
work (item 1) lands, promote both bases to `workflow.db.errors`; have `ContentServiceError`'s
subtree inherit from the neutral bases so `except ContentServiceError` still catches everything.
Pair with: amend **ADR-0007** with a note — "service-layer error taxonomies introduced in a
feature module are provisional; generic bases graduate to `workflow.db.errors` once a second
consumer appears." Also, when this lands, reconsider whether `BibLinkNotFound` should join
`EntityNotFoundError` (python-reviewer LOW-1; architect L1 argues keep it standalone — a missing
*link* is not a missing *entity* — resolve the disagreement then).

### 6. (from v1.16.0 review) Extract a `workflow.graph.node_ids` helper — LOW
The `note:<int>` id convention is parsed in several spots in `graph/cli.py`
(`_fetch_note_rows` split, `startswith("note:")` guards ×4, `_note_path`). Extract
`is_note(node_id)` + `parse_note_id(node_id) -> int | None` before a third consumer
appears. Architect finding, v1.16.0 reviewer-esquema.

### 7. (from v1.16.0 review) Resolve the `relation_type=edge_type` overload — MEDIUM
`graph neighbors --json` emits `relation_type=<GraphEdge.edge_type>` (knowledge-graph
type: `link`/`citation`), which collides with the NoteEdge semantic `relation_type`
(`supports`/`contradicts`) the `notes edges` picker uses. Decide: rename the key to
`edge_type`, or add a `"graph":"knowledge"` discriminator to the envelope. Update
**ADR-0017** + all three surfaces (cli.py, test, Lua picker) together.

### 8. (from v1.17.0 review) Move the bulk-import engine out of `topic/` — HIGH (follow-up)
`workflow/topic/bulk_import.py` is a cross-domain composition root (topic+content+concept)
living in a leaf module that now imports `add_content`/`add_concept` from sibling domains.
Before a 2nd import surface (bib, notes) lands, extract to a neutral `workflow/importer/`
and expose `workflow import` (the verb imports a hierarchy rooted at DisciplineArea, not a
topic). Keep `topic import` as a deprecation alias. Architect finding, v1.17.0.

### 9. (from v1.17.0 review) Shared duplicate-lookup helpers — MEDIUM
`bulk_import.py` re-implements the uniqueness predicates (`topic (da_id, serial)`,
`content (topic_id, name)`) that already live inside `add_topic`/`add_content`. If a guard's
key changes (e.g. content keyed on a slug), the importer's "find existing" query silently
drifts. Extract `get_topic_by_serial` / `get_content_by_name` into the services and use them
in both the guard and the importer.

### 10. (from v1.17.0 review) Document concept global-skip — LOW
A concept `code` reused under a *different* content is silently skipped (global code
uniqueness), not re-linked. Document in `workflow topic import --help` + the YAML template.
Also revisit per-row SAVEPOINTs (ADR-0018) if the global engine adopts the pysqlite BEGIN recipe.
