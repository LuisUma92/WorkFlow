## Phase 4 — Zettelkasten main-topic bundle (multi-day, biggest)

`tasks/requests/2026-05-04-zettelkasten-main-topic-bundle.md` — Phase A/B/C
unstarted; depends on ITEP-0009 (currently "Implemented (partial)").

Split into:

- ✅ **4A** (commit `2f04052`, 2026-05-27): ITEP-0009 flipped to Implemented after triage — WorkFlow-repo scope was already shipped; Part III items live in the ~/01-U workspace. No code changes, only ADR labels.
- **4B** (NEW, user-decided 2026-05-27): re-root `Topic` at `DisciplineArea` + new `MainTopicSyllabus(main_topic_id, topic_id, week_no, order_no)` join. Migration `0011_topic_root_discipline_area`. Spec: `tasks/requests/2026-05-27-topic-reroot-discipline-area.md`. Target: `v1.11.0`.

  Rationale: current schema forces `Content`/`BibContent`/`Concept`/`Exercise` links into a single MainTopic, breaking cross-project reuse. Topic-as-canonical-field-chapter (rooted at DisciplineArea) restores reuse; per-iteration ordering moves into the syllabus join. Live DB has 0 topic rows → structural-only migration.

- **4C** (was 4B): Phase A of `tasks/requests/2026-05-04-zettelkasten-main-topic-bundle.md` — notes↔main_topic frontmatter binding.

  **Re-evaluation post-4B**: Binding notes to `MainTopic` (project tag) remains meaningful — it tags a note by project iteration. After 4B, the OTHER axis (note↔Topic canonical chapter) becomes equally important and reusable. Scope question: do both bindings ship in 4C, or just `main_topic`? Recommend: ship `main_topic` first per the original request; add `topic` as a follow-up if the bundle request gets amended.

- **4D** (was 4C): Phase B — main_topic FK propagation through validation.

  **Re-evaluation post-4B**: Was "validate that note.main_topic FK resolves to a real MainTopic". After 4B, the `Concept.main_topic` chain via `content.topic.main_topic` no longer exists — MainTopic for a concept is project-context-dependent. Strict-main-topic check on concepts must reach MainTopic via `MainTopicSyllabus` lookup OR be reduced to strict-discipline-area check (cheaper, no project context needed). Recommend reducing to discipline-area check for concepts; main-topic FK validation still applies to notes (they own a direct `main_topic` field).

- **4E** (was 4D): Phase C — main_topic-aware graph filters.

  **Re-evaluation post-4B**: With `MainTopicSyllabus`, "main_topic-aware filter" becomes "filter graph to nodes reachable from a MainTopic via its syllabus". Implementation: traverse `MainTopic → MainTopicSyllabus → Topic → Content → {BibContent, Concept, Exercise}`. Possibly add an inverse filter "what canonical Topics aren't yet in any MainTopicSyllabus" (curriculum gap report).

Estimate 1–2 weeks if all phases land. Tag progression `v1.11.0` →
`v1.12.0` → `v1.13.0` → `v1.14.0`.
